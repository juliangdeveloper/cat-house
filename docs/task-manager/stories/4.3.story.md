### Architecture: FastAPI OpenAPI Integration

**Automatic OpenAPI Generation:**

FastAPI automatically generates OpenAPI 3.0 specification from:
- Endpoint decorators (`@router.post`, `@router.get`)
- Pydantic models (request/response schemas)
- Type hints (parameter types, return types)
- Docstrings (endpoint descriptions)

**Customizing OpenAPI Metadata:**

```python
# app/main.py
app = FastAPI(
    title="Task Manager API",
    description="""
    Task Manager API for Cat House Platform integration.
    
    Uses Universal Command Pattern for all task operations.
    All commands are sent to POST /execute endpoint.
    
    ## Authentication
    Requires X-Service-Key header for all protected endpoints.
    
    ## Available Actions
    - create-task: Create a new task
    - list-tasks: List user's tasks
    - get-task: Get single task by ID
    - update-task: Update existing task
    - delete-task: Delete task by ID
    - get-stats: Get task statistics
    """,
    version="0.1.0",
    docs_url="/docs",
    redoc_url="/redoc",
    openapi_url="/openapi.json",
)
```

**Adding Security Schemes:**

```python
# Define security scheme
app = FastAPI(
    # ... other metadata ...
    swagger_ui_init_oauth={
        "usePkceWithAuthorizationCodeGrant": True
    },
)

# Add to OpenAPI schema
def custom_openapi():
    if app.openapi_schema:
        return app.openapi_schema
    
    openapi_schema = get_openapi(
        title=app.title,
        version=app.version,
        description=app.description,
        routes=app.routes,
    )
    
    # Add security scheme
    openapi_schema["components"]["securitySchemes"] = {
        "ServiceKey": {
            "type": "apiKey",
            "in": "header",
            "name": "X-Service-Key",
            "description": "Service API key for Cat House integration"
        },
        "AdminKey": {
            "type": "apiKey",
            "in": "header",
            "name": "X-Admin-Key",
            "description": "Admin key for service key management"
        }
    }
    
    app.openapi_schema = openapi_schema
    return app.openapi_schema

app.openapi = custom_openapi
```

[Source: FastAPI documentation - https://fastapi.tiangolo.com/advanced/extending-openapi/]

### Architecture: Pydantic Model Examples

**Adding Examples to Models:**

```python
# app/commands/models.py
class CommandRequest(BaseModel):
    action: str = Field(..., description="Command action name")
    user_id: str = Field(..., description="User ID from Cat House JWT")
    payload: dict = Field(..., description="Action-specific payload")
    
    class Config:
        schema_extra = {
            "examples": [
                {
                    "action": "create-task",
                    "user_id": "user_123",
                    "payload": {
                        "title": "Buy cat food",
                        "description": "Whiskers needs more treats",
                        "status": "pending",
                        "due_date": "2025-11-15T10:00:00Z"
                    }
                },
                {
                    "action": "get-stats",
                    "user_id": "user_123",
                    "payload": {}
                }
            ]
        }

class CommandResponse(BaseModel):
    success: bool = Field(..., description="Whether command succeeded")
    data: dict = Field(..., description="Response data (structure depends on action)")
    error: Optional[str] = Field(None, description="Error message if success=false")
    timestamp: datetime = Field(default_factory=datetime.utcnow, description="Response timestamp")
    
    class Config:
        schema_extra = {
            "example": {
                "success": True,
                "data": {
                    "total_tasks": 10,
                    "pending_tasks": 3,
                    "in_progress_tasks": 2,
                    "completed_tasks": 5,
                    "completion_rate": 50.0,
                    "overdue_tasks": 2
                },
                "error": None,
                "timestamp": "2025-11-13T12:00:00Z"
            }
        }
```

**Why Examples Matter:**

- Swagger UI shows examples in request/response forms
- Developers can copy-paste examples to test API
- Documentation is self-contained (no external docs needed)
- Examples serve as integration tests (verify examples work)

[Source: Pydantic documentation - https://docs.pydantic.dev/latest/usage/schema/#schema-customization]

### Architecture: Error Response Documentation

**Documenting Error Formats:**

Create error response models to document error structures:

```python
# app/models/errors.py
from pydantic import BaseModel, Field
from typing import List, Optional

class ValidationError(BaseModel):
    """Validation error detail"""
    loc: List[str] = Field(..., description="Error location (field path)")
    msg: str = Field(..., description="Error message")
    type: str = Field(..., description="Error type")

class ValidationErrorResponse(BaseModel):
    """422 Validation Error Response"""
    detail: List[ValidationError] = Field(..., description="List of validation errors")
    
    class Config:
        schema_extra = {
            "example": {
                "detail": [
                    {
                        "loc": ["body", "action"],
                        "msg": "field required",
                        "type": "value_error.missing"
                    }
                ]
            }
        }

class ErrorResponse(BaseModel):
    """Standard error response"""
    detail: str = Field(..., description="Error message")
    
    class Config:
        schema_extra = {
            "example": {"detail": "Invalid service key"}
        }
```

**Use in Endpoint Responses:**

```python
@router.post(
    "/execute",
    response_model=CommandResponse,
    responses={
        400: {"model": ErrorResponse, "description": "Invalid action"},
        401: {"model": ErrorResponse, "description": "Authentication failed"},
        422: {"model": ValidationErrorResponse, "description": "Validation error"}
    }
)
async def execute_command(...):
    pass
```

### Architecture: ReDoc Alternative

**ReDoc vs Swagger UI:**

FastAPI provides two documentation UIs by default:

1. **Swagger UI (/docs):**
   - Interactive "Try it out" feature
   - Execute API requests directly from browser
   - Best for API testing and exploration

2. **ReDoc (/redoc):**
   - Clean, professional layout
   - Better for reading documentation
   - No interactive features (read-only)
   - Better for public-facing docs

**When to Use Each:**

- **Development:** Swagger UI for testing
- **Documentation:** ReDoc for sharing with Cat House team
- **Both:** Include links to both in README

[Source: FastAPI documentation - https://fastapi.tiangolo.com/advanced/extending-openapi/#alternative-api-docs-ui-redoc]
